
## 项目的结构
良好的项目管理是整个项目的好的开始,原项目的结构非常清晰.

    - model 模型定义
    - trainer 不同的训练代码
    - dataset 训练的数据集类型
    - scripts 转换,评估等脚本

我们这个项目对其进行了一些修改,从flat layout改成了src layout, 并加入了一些其它文件夹来适配不同功能.

    - src 项目的核心模块化代码,包含模型定义,训练代码,数据集定义等
    - scripts 转换,评估等脚本
    - notes(docs) 项目的文档,包含一些笔记和教程
    - minimind_upstream 上游代码,使用submodule管理,作为参考
    - configs 配置文件
    - tests 测试代码
    

## 项目管理工具
推荐使用UV作为项目的管理工具,并使用src layout.

```bash
    # 同步项目
    uv sync 
    
    # 以开发模式安装包
    uv pip install -e .
```

UV是一个现代的Python包管理工具,它支持更快的安装和更好的依赖解析.但是它的接口有点复杂，它有自己的包管理接口，也同时兼容pip接口。两者的区别是uv add会把包添加到pyproject.toml的dependencies中，而uv pip install则不会修改pyproject.toml,只是安装包而已.所以如果你想安装一个包并且让它成为项目的一部分,应该使用uv add.如果你只是想临时安装一个包来测试一下,可以使用uv pip install.

```bash
    # 使用UV的包管理接口安装包
    uv add package_name
    uv remove package_name
    uv add -r requirements.txt
    
    # 使用uv pip接口安装包 (只需要把pip 换成uv pip就可以了)
    uv pip install package_name
    uv pip install -r requirements.txt
    uv pip uninstall package_name
```

这部分UV确实有一点混乱，pip 和uv自己的命令都可以兼容。

``` bash
    # 显示当前环境的包和版本
    uv tree
    uv pip list
    uv pip show package_name 
```

## UV配置Pytorch
Pytorch的安装有一些特殊性，目前Pytorch不支持版本号加后缀来选择cuda版本"torch==2.2.0+cu121"，而是通过指定不同的源（index）来安装不同版本的pytorch.所以如果你直接使用默认的uv add torch 会从Pypi安装CPU版本的Pytorch.

如果需要安装CUDA版本需要在pyproject.toml里面对torch包指定单独的源.就像pytorch官网使用pip安装时指定index一样.可以参考：
[官方文档](https://docs.astral.sh/uv/guides/integration/pytorch/#configuring-accelerators-with-environment-markers)

本项目使用了uv extra机制来适配CPU和CUDA平台.
```bash

  # 安装CPU版本的Pytorch
  uv sync --extra cpu

  # 安装CUDA版本的Pytorch
  uv sync --extra cuda

```

注: uv的extra机制有一点混乱,上面的官方示例有一些问题,使用`uv sync --extra flag` 安装包,之后每次添加或者移除包执行`uv add/remove` 或者`uv sync`会导致extra对应的包被移除,需要重新执行一遍`uv sync --extra flag` 来安装对应的包.

解决办法是再dependencies里面添加torch cpu 和 CUDA 共通的依赖版本.如下面案例所示.

注: UV extra的解析机制,UV 会做这么几件事情: 

1. Configuration Validation（配置校验）
   - 检查 extras 是否互斥
   - 检查 conflicts 是否自洽
   - 检查 pyproject.toml 是否结构正确
   - 检查 index 配置是否冲突

2. Dependency Resolution（依赖解析）
   - 合并依赖
   - 合并 extras（已启用的）, dependencies中的依赖会被默认启用,但是extra中的默认不启用,只有加入extra flag才会启用.
   - 选择版本
   - 生成锁定图
   - 类似的命令 uv pip  compile pyproject.toml

3. Installation（安装）
   - 下载 wheel/sdist
   - 安装到 .venv

`uv remove` 
  会修改 pyproject.toml uv 必须确保修改后的项目配置是“合法的”所以会先执行 Configuration Validation这一步会检查 extra conflicts即使 extras 没启用，也会检查 conflicts 是否自洽,所以会报错.但是这个报错又不影响安装.

`uv sync` 
  不修改 pyproject.toml 
  uv 不需要重新验证配置
  uv 默认不启用 extras
  所以不会触发 conflicts 检查
  直接进入 dependency resolution → installation
  
  但是dependencies中是最小依赖,所以不会重新安装,而如果dependencies中没有torch,默认的uv sync 不会解析extra,就会移除extra中的torch,所以需要重新执行`uv sync --extra flag` 来安装对应的包.

(这实在是有点混乱了,也许以后会改,总而言之下面的配置是目前最佳实践)

```toml
dependencies = [
    "swanlab>=0.6.13",
    "transformers>=4.57.1",
    "torch>=2.9.0",
    "torchvision>=0.24.0",
]

[tool.uv]
conflicts = [
  [
    { extra = "cpu" },
    { extra = "cuda" },
  ],
]

[project.optional-dependencies]
cpu = [
    "torch>=2.9.0",
    "torchvision>=0.24.0",
]
cuda = [
    "torch>=2.9.0",
    "torchvision>=0.24.0",
]

[[tool.uv.index]]
name = "pytorch-cu126"
url = "https://download.pytorch.org/whl/cu126"
explicit = true
[[tool.uv.index]]
name = "pytorch-cu126_c"
url = "https://mirrors.nju.edu.cn/pytorch/whl/cu126"
explicit = true


[[tool.uv.index]]
name = "pytorch-cpu"
url = "https://download.pytorch.org/whl/cpu"
explicit = true

[tool.uv.sources]
torch = [
  { index = "pytorch-cu126_c", extra = "cuda" },
  { index = "pytorch-cpu", extra = "cpu" },
]
torchvision = [
  { index = "pytorch-cu126_c", extra = "cuda" },
  { index = "pytorch-cpu", extra = "cpu" },
]

```
